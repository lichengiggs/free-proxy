# Free Proxy 技术重构方向：轻量稳定方案

## 1. 这次重构要解决什么

这个项目当前最核心的问题不是“还能不能继续加功能”，而是：

- agent 场景不稳定，尤其是 OpenClaw / coding agent 这种持续对话场景
- 过去完全自研的路由和 fallback 逻辑，AI 容易修很多轮仍然修不稳
- 你希望保留学习价值，但不能再把复杂度全压回自己身上

因此，这次技术方案的目标不是做一个更大的网关，而是做一个**足够轻、足够稳、足够可理解**的稳定层。

## 2. 需求收敛后的真实目标

### Must-have

- **稳定可用** 优先于功能数量
- agent 场景优先，网页 UI 只是辅助验证入口
- 前台仍然可以展示“很多模型可选”的产品感
- 后台必须自动化处理主备切换
- 主备策略优先看 **成功率** 和 **连续可用时长**
- 模型名字、provider 数量、可用额度都不假设固定

### Nice-to-have

- 前台保留多个推荐模型入口，增强“免费额度很多”的感知
- 后续再逐步引入更聪明的模型评分或质量路由
- 未来支持更多 provider 时，不需要改核心逻辑

### 被否定的假设

- “额度大就一定稳定”不成立
- “模型越多越稳”不成立
- “每次请求都重新挑最优模型”不适合当前目标

## 3. 推荐的技术方案

我建议把方案固定为下面这一套：

- `httpx`：负责上游请求、异步调用、流式读取
- `tenacity`：负责重试、指数退避、抖动
- `aiolimiter`：负责请求节流和速率保护
- `cachetools`：负责短期健康状态、失败计数、热备冷却信息
- 你们自己的路由逻辑：负责候选排序、主备选择、错误分类、结果收敛

这是一套**小而完整**的组合：

- 传输交给成熟库
- 重试交给成熟库
- 节流交给成熟库
- 状态缓存交给成熟库
- 路由决策保留在你们自己的代码里

这样既保留学习价值，又能明显降低维护成本。

## 4. 为什么不是 LiteLLM 作为主核心

LiteLLM 能做很多事，但对你们当前阶段来说偏重。

你现在最需要的是：

- 看到每次失败为什么发生
- 能快速定位 agent 场景为什么中断
- 让 AI 改代码时尽量只改局部
- 保持项目结构清晰、可解释

而 LiteLLM 会把很多能力一起打包进去：

- 路由
- 负载均衡
- fallback
- 冷却
- 速率限制
- Redis 状态共享

这些能力不是不好，而是**超出当前最小够用范围**。对这个项目来说，先做轻量稳定层更合适。

## 5. 主备模型不硬编码的设计原则

这是这次方案的关键。

### 原则

- 不硬编码具体模型名
- 不假设 provider 数量固定
- 不假设某个 API 会永远可用
- 不假设用户总会选到最稳的模型

### 推荐做法

把“主备”理解成**角色**，不是固定名称：

- 主 provider：当前综合评分最高、连续可用时长最长的候选
- 热备 provider：当前第二优的候选
- 如果主 provider 连续失败 N 次，就切到热备
- 如果热备也失败，再从候选池重新计算

也就是说，主备不是写死的 `A -> B`，而是运行时动态选择出来的。

### 候选来源

候选不要只依赖手写模型名，而是组合这些来源：

- 配置里已启用的 provider
- provider 的 `model_hints`
- 运行时 `/models` 探测结果（如果可用）
- 历史健康状态缓存
- 最近成功调用的 provider/model

这样即使模型名字变了、API 数量变了，也不需要重构核心路由。

## 6. 路由逻辑应该怎么做

### 推荐流程

1. 读取已配置 provider 列表
2. 为每个 provider 获取可用模型候选
3. 结合 `cachetools` 中的健康状态和失败计数打分
4. 选出当前最稳的 provider/model 作为主 provider
5. 选出下一名作为热备
6. 请求优先走主 provider
7. 连续失败达到阈值后切换到热备
8. 主 provider 恢复后再重新评估

### 评价信号优先级

当前应优先看：

- 成功率
- 连续可用时长
- 最近失败次数
- 响应是否稳定
- 是否出现超时、空结果、流式中断

不要把“模型强弱”放在第一优先级。对你们这个项目来说，**稳定比聪明更重要**。

## 7. 每个库在方案里的职责

### `httpx`

负责真正的上游通信。

适合你们的原因：

- async 支持成熟
- 适合流式读取
- 轻量，不会把架构推得太重
- 非常适合 Python 原生项目

### `tenacity`

负责失败重试和退避。

适合你们的原因：

- 你们的问题不是“要不要重试”，而是“怎么重试才稳”
- 可以避免自己写复杂 retry loop
- 能处理短暂网络抖动和偶发失败

### `aiolimiter`

负责请求节流。

适合你们的原因：

- 免费 API 场景很容易误打爆某个 provider
- 节流可以避免把“请求太快”误判成“provider 坏了”
- 这是 agent 场景稳定性的基础设施

### `cachetools`

负责短期状态缓存。

适合你们的原因：

- 可以缓存 provider 的健康状态
- 可以记录连续失败次数
- 可以实现热备冷却窗口
- 不需要一开始就引入 Redis

### 你们自己的路由逻辑

只保留真正属于产品决策的部分：

- 候选排序
- 主备选择
- 错误分类
- 切换阈值
- 稳定性埋点

这一层要尽量短、直、可测试。

## 8. 为什么这个方案更适合学习型项目

这个方案比“大网关方案”更适合你们，原因很直接：

- 你能看懂每一层在做什么
- AI 改代码时更容易局部正确
- 出问题时更容易定位
- 学到的是系统设计，而不是只会配框架
- 后续分享给朋友时，故事也更清楚

对你来说，最重要的是：**项目稳定了，学习意愿才会持续。**

## 9. 不建议现在做的事

- 不要把 LiteLLM 当主核心
- 不要先上 Redis
- 不要做复杂的动态路由策略
- 不要把每次请求都做全局最优搜索
- 不要硬编码模型名和 provider 数量

这些都不是当前 MVP 的重点。

## 10. 推荐的最小实现边界

### MVP 只做这些

- 一个统一 OpenAI 兼容入口
- 一个主 provider
- 一个热备 provider
- 连续失败 N 次才切换
- provider 健康状态短期缓存
- 请求节流和重试
- 失败时自动降级

### 暂时不做这些

- 复杂的多级调度
- Redis 分布式状态
- 模型成本优化
- 大规模并发调度
- 过多的手工优先级规则

## 11. 最终技术栈建议

### 必选

- `httpx`
- `tenacity`
- `aiolimiter`
- `cachetools`
- 你们自己的路由逻辑

### 可选后置

- `pydantic-settings`：当配置项明显变多时再引入
- `pybreaker`：当你们需要更明确的熔断语义时再引入

### 暂不采用

- LiteLLM 作为底层核心
- Redis
- 复杂动态路由系统

## 12. 最终结论

如果按最小够用原则，这个项目最适合的路线不是“做一个更大的聚合器”，而是：

- 前台保留丰富选择
- 后台只做主备自动切换
- 候选池动态生成，不硬编码模型名
- 用成熟小库解决网络、重试、限流、状态缓存
- 保留你们自己的路由决策和产品外壳

这条路线最适合你现在的目标：

- 稳定可用
- 能继续学习 vibecoding
- 能长期迭代
- 能对朋友和社交媒体分享
