# free-proxy 产品需求文档

## 1. 背景

当前项目在 OpenClaw 这类 agent 场景下，常见失败模式有两类：

1. 单次请求上下文过大，免费模型即使可列出、可发现，也在真实调用时高频超时、429、503 或直接失败。
2. fallback 仅按静态排序遍历候选模型，缺少基于近期失败的熔断和动态降权，导致短时间内重复命中同一批不可用模型，延迟高、成功率低。

补充约束：fallback 的主顺序应先跨 provider 轮询，再在 provider 内按模型排序选择；如果某个 provider 已经持续失败，继续在该 provider 内穷举模型通常只会重复失败，因此应尽快切换到其他 provider，以减少无效等待时间。

日志已经体现出：

- 服务能正常启动，但一次失败会连续 fallback 多个免费模型。
- 已发现的免费模型基本都被遍历到，但“遍历到”不等于“适合当前请求”。
- 问题重点不是模型列表不全，而是请求负载与路由策略不匹配。

补充目标：页面首次打开时应主动拉取并展示可用模型列表，避免依赖刷新后才看到结果，减少用户误以为“没有免费模型”的困惑。

补充约束：如果某些 provider 长期没有模型进入候选集，需要优先排查上下文门槛是否过于保守；上下文阈值可以按 provider 或模型的官方能力进行微调，但仍需保持保守默认值，避免把明显不适合的大请求发给低上下文模型。

本次需求目标是：在不伪装官方客户端、不绕过供应商限制的前提下，让免费模型在大上下文 agent 请求中尽可能跑通，并尽量减少无效 fallback。

## 2. 目标

### 2.1 核心目标

1. 增加“上下文裁剪”能力，在请求过大时主动降载。
2. 增加“失败熔断 + 动态降权”能力，在 provider 或 model 不稳定时自动避让。
3. 优先提升 OpenClaw 这类长上下文、多轮、工具调用场景下的成功率与平均响应时间。

### 2.2 成功标准

1. 相同输入下，超长请求不再直接把全部原始消息无脑转发给免费模型。
2. 单个模型连续失败后，不会在短时间内被重复优先尝试。
3. fallback 链在失败场景下明显缩短无效尝试次数。
4. 日志能清晰说明：请求是否被裁剪、为何跳过某模型、模型为何降权。

## 3. 非目标

以下内容不属于本期：

1. 不实现伪造官方客户端 Header、User-Agent、平台指纹。
2. 不尝试绕过供应商风控、配额或免费限制。
3. 不引入复杂的长期记忆系统、向量数据库或历史会话存储。
4. 不做前端大改版，优先完成后端能力。

## 4. 用户场景

### 场景 A：OpenClaw 发起大上下文任务

用户通过 `free-proxy` 访问免费模型，请求中包含大量历史消息、工具输出和长 system prompt。系统需要先判断请求是否超预算，再决定：

- 原样发送
- 裁剪历史后发送

目标：尽量让请求“能跑起来”，而不是把所有内容原样塞给免费模型导致整批失败。

### 场景 B：某些免费模型短时间内大面积失败

用户连续发起多次请求，某些 provider/model 已经出现 429、503、超时或连续网络错误。系统需要自动把这些模型临时降级，不再继续高频命中。

目标：避免一次失败后继续用同一套坏模型链路浪费 10~30 秒。

## 5. 功能需求

## 5.1 请求预算管理

系统必须在转发到上游前，对每次聊天请求进行预算评估。

### 输入

- `model`
- `messages`
- `tools` / `functions` / tool schema

### 输出

- 估算输入 token 数
- 是否需要裁剪（`shouldTrim`）

### 判断逻辑

- 估算输入 token > 全局 token 阈值 → `shouldTrim = true`
- 阈值默认值由环境变量 `REQUEST_BUDGET_DEFAULT_INPUT_TOKENS` 控制，保守默认值 10000
- 后续可支持 per-model / per-provider 覆盖，第一期只做全局

### 要求

1. 必须提供统一的 token 估算逻辑，即使不是精确 tokenizer，也要在项目内保持一致。
2. 估算结果必须写入日志，至少包含：`model`、`estimated_input_tokens`、`message_count`、`tool_count`。
3. 预算评估必须发生在真正调用上游之前，而不是失败之后再补救。

## 5.2 上下文裁剪

当请求预算超阈值时，系统必须裁剪消息后再发送。

### 裁剪策略

采用简单截断：只保留系统关键消息，删除中间历史。

1. 保留 system message（第一条 system 角色消息）。
2. 保留最后一轮用户消息。
3. 可选保留最后一个 assistant 回复（如有）。
4. 删除其余所有中间消息。

不做摘要、不做 tool 输出压缩。第一期目标是"让请求跑起来"，不是"保留完整对话上下文"。

### 行为要求

1. 裁剪后消息列表不能为空，必须至少包含一条用户消息。
2. 裁剪必须是"保关键、删历史"，不能直接删掉最后一轮用户问题。
3. 需要在日志中记录：
   - 原始消息数 / 裁剪后消息数
   - 原始估算 token / 裁剪后估算 token

## 5.3 模型级预算策略

不同模型应允许不同的预算阈值，但第一期只做全局配置，不做复杂自动学习。
### 要求

1. 必须支持全局默认阈值。
2. 后续支持按 provider 或 model 覆盖阈值。
3. 当模型上下文能力未知时，按保守阈值处理。
4. 如果 `models.dev` 或其他模型字典已经给出 provider/model 的 `input_limit` / `context_limit`，应优先用该值覆盖默认阈值，避免对已知大上下文模型过度保守。

### 建议配置项

- `REQUEST_BUDGET_DEFAULT_INPUT_TOKENS`（全局 token 上限）
- 后续可加：per-model / per-provider 覆盖表

## 5.4 失败分类

系统必须把失败按类型分类，而不是全部都当成一个“失败”。

### 最低分类要求

- `rate_limit`：429、明确额度限制
- `auth_error`：401 / 403
- `provider_unavailable`：503 / 网关不可用
- `timeout`：请求超时
- `network_error`：网络异常
- `bad_request`：400 / 请求体不兼容
- `unknown_error`

分类结果必须进入日志，并为后续熔断与降权提供输入。

## 5.5 失败熔断

当某个模型或 provider 在短时间内连续失败时，系统必须临时跳过它。

### 模型级熔断

1. 同一模型在滑动窗口内连续失败达到阈值后，进入 `open` 状态。
2. `open` 状态下该模型不再参与优先尝试，只能在冷却期后进入 `half_open`。
3. `half_open` 状态允许少量试探请求；若成功则恢复，若失败则重新熔断。

### Provider 级熔断

1. 同一 provider 下多个模型连续失败时，可触发 provider 级降级。
2. provider 级熔断后，该 provider 的模型整体后移，而不是完全永久禁用。

### 要求

1. 熔断状态必须带过期时间。
2. 熔断状态必须可记录到持久层，重启后不能全部丢失。
3. 熔断必须区分失败类型：
   - `auth_error` 应立即强熔断，并提示用户检查 key
   - `rate_limit` 应按冷却时间处理
   - `timeout` / `provider_unavailable` 应进入短期熔断

## 5.6 动态降权

即使未达到熔断阈值，近期表现差的模型也必须在排序中被降权。

### 降权输入因子

1. 最近成功率
2. 最近连续失败次数
3. 最近平均响应时长
4. 最近失败类型（429、503、timeout 权重更高）
5. 当前请求风险等级（大请求时，小上下文或慢模型要进一步降权）

### 行为要求

1. 动态降权只影响排序，不直接删除模型。
2. 静态评分仍保留，但动态评分优先级更高。
3. 每次 fallback 前都应基于最新状态重新排序。
4. 日志需打印前若干个候选模型的最终分数构成，至少包括：`static_score`、`dynamic_penalty`、`final_score`。

## 5.7 Fallback 执行策略

fallback 不应再只做“从头到尾穷举”。

### 新规则

1. 先根据请求风险等级筛掉明显不适合的大请求模型。
2. 再按动态分数排序。
3. 遇到已熔断模型，默认跳过并写日志。
4. 同一 provider 短时间连续失败后，优先切换到其他 provider。
5. 必须保留一个最终兜底模型，但要避免在明显不可能成功的模型上浪费过多尝试。
6. Provider 内部的模型排序应结合静态评分和动态健康度，但在 provider 整体异常时，应优先换 provider，而不是在同一 provider 内继续穷举。

### 限制

1. 单次请求的 fallback 最大尝试数必须可配置。
2. 达到最大尝试数后立即返回明确错误，不再继续长链遍历。

建议配置项：

- `FALLBACK_MAX_ATTEMPTS`
- `MODEL_CIRCUIT_BREAKER_FAILURE_THRESHOLD`
- `MODEL_CIRCUIT_BREAKER_COOLDOWN_MS`
- `PROVIDER_CIRCUIT_BREAKER_FAILURE_THRESHOLD`

## 5.8 日志与可观测性

必须补齐结构化日志，便于后续判断到底是"请求过大"还是"供应商不可用"。

### 每次请求至少记录

1. 入口信息：请求模型、消息数、工具数、估算 token。
2. 预算动作：是否裁剪、裁剪前后 token。
3. 路由决策：候选链长度、最大尝试数、跳过原因。
4. 失败细节：provider、model、失败分类、HTTP 状态码、耗时。
5. 排序依据：动态降权结果、熔断状态。

### 管理接口

后续应预留只读接口或调试接口，用于查看：

- 当前熔断状态
- 最近失败统计

本期允许先只写日志，不强制做 UI。

## 6. 约束与原则

1. 所有能力必须在当前项目架构内实现，优先落在 `src/server.ts`、`src/fallback.ts`、`src/rate-limit.ts` 或新增轻量模块。
2. 不引入大型外部依赖，不做过度工程化。
3. 裁剪后消息不能为空，必须保留至少一条用户消息。
4. 文案和错误信息要对普通用户友好，不暴露实现细节。

## 7. 验收标准

### 功能验收

1. 构造超长消息请求时，服务日志能明确显示发生了裁剪。
2. 构造某模型连续 429/503 的场景时，该模型会在后续请求中被跳过或降权。
3. 在多模型失败场景下，日志中能看出排序变化和跳过原因。
4. 错误响应中不再只给"全部失败"的黑盒结果，而是能从日志中定位失败根因。

### 测试验收

至少补充以下测试：

1. 请求预算评估测试
2. 上下文裁剪测试
3. 模型级熔断状态流转测试
4. provider 级降级测试
5. 动态降权排序测试
6. fallback 最大尝试数测试

## 8. 实施优先级

### P0

1. 请求 token 估算
2. 上下文裁剪（简单截断）
3. 失败分类
4. 模型级熔断
5. 动态降权排序
6. 关键日志

### P1

1. provider 级熔断
2. per-model / per-provider 阈值覆盖
3. 调试接口

### P2

1. 更精细的模型预算画像
2. 基于历史成功率的长期自适应参数

## 9. 一句话方案

让 `free-proxy` 从"静态穷举免费模型列表"升级成"先裁剪请求，再按实时健康度选择模型"的代理层。
